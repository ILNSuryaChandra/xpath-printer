## Codebase Patterns

- Libraries (html2canvas, jsPDF) are bundled locally in `libs/` and injected via `chrome.scripting.executeScript` in the background service worker **before** messaging the content script.
- Content script is registered via `manifest.json` `content_scripts` array (not injected programmatically), so it is always present and ready to receive messages.
- The popup sends a message to the background worker with the tab ID; it does NOT talk to the content script directly.

## What was implemented

## Learnings for future iterations

Patterns discovered:
- Chrome MV3 does not allow eval/remote scripts; all libs must be bundled locally in the extension directory and declared in web_accessible_resources.
- Service workers (background.js) in MV3 cannot use XMLHttpRequest — use fetch instead.
- chrome.scripting.executeScript requires the "scripting" permission AND the tab URL must match host_permissions or activeTab must be granted.
- The popup auto-closes after activation so the user can see and interact with the page — this is intentional UX.

Gotchas encountered:
- sips on macOS warns "Output file suffix should be jpg" when outputting PNGs — this is harmless; the files are valid PNGs.
- html2canvas CDN URL had a redirect; curl -sL (follow redirects) was needed.
